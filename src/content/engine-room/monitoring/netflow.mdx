## Overview

SNMP tells you how many bytes crossed an interface. That's useful but incomplete — it's like knowing the total weight of all the packages that left your warehouse without knowing where any of them went.

**NetFlow** (Cisco, 1996) fills that gap by having routers track individual traffic flows and export records to a collector. A flow record answers: which source IP talked to which destination IP, on which port, using which protocol, for how long, and how many bytes. The result is a complete conversation log of everything passing through your network.

**IPFIX** (IP Flow Information Export, RFC 7011, 2013) is the IETF standard that succeeded NetFlow v9. Functionally identical, vendor-neutral, and now the preferred term in new deployments.

---

## What a Flow Is

A flow is a unidirectional sequence of packets sharing the same **5-tuple**:

1. Source IP address
2. Destination IP address
3. Source port
4. Destination port
5. IP protocol (TCP=6, UDP=17, ICMP=1)

Plus: router input interface, IP ToS byte (DSCP), and optionally BGP AS numbers and next-hop.

Each direction of a TCP connection is a separate flow. A complete TCP session between two hosts generates two flow records.

**Flow expiry conditions**:
- **TCP FIN/RST**: connection teardown detected — export immediately
- **Idle timeout** (default 15 seconds): no packets for this flow in 15s — export and remove
- **Active timeout** (default 30 minutes): long-lived flow (e.g. a file transfer) — export periodically so data doesn't sit in cache for hours
- **Cache full**: oldest flows evicted and exported

## NetFlow Versions

| Version | Format | Notes |
|---------|--------|-------|
| **v1** | Fixed 16 fields | Original, rarely seen |
| **v5** | Fixed 7-key fields | Most common on classic IOS. IPv4 only. No IPv6, no MPLS. |
| **v9** | Template-based | Flexible — any fields. Supports IPv6, MPLS, BGP AS. Templates sent periodically. |
| **IPFIX** | Template-based (IETF) | RFC 7011. Based on NetFlow v9. Vendor-neutral. Preferred for new deployments. |

**Why v9/IPFIX matter**: Fixed formats can't express IPv6 addresses, MPLS labels, or application IDs (NBAR). Template-based formats define which fields are in a record before sending the data — the collector uses the template to decode subsequent records.

## sFlow — The Alternative Approach

sFlow (RFC 3176) takes a different approach: **random packet sampling** rather than complete flow tracking.

- Every Nth packet (configurable — commonly 1:1000 or 1:512) is sampled
- The full packet header (and optionally payload) is sent directly to the collector
- The collector extrapolates traffic volume from the sample

**Advantages of sFlow**:
- Works at wire speed on any hardware — no per-flow state in the router
- Simpler to implement — commodity switches support it; not all support NetFlow
- Can capture layer 2 information (VLANs, MAC addresses)

**Disadvantages**:
- Results are **statistical estimates** — fine for trend analysis, wrong for billing or forensics
- Short flows (DNS queries, connection attempts) may not be sampled at all
- Higher collector load at high sampling rates

---

## Sampling in NetFlow

On high-speed links (10G+), a router can't maintain a flow cache entry for every packet — the cache would exhaust memory and the CPU would be overwhelmed. **Sampled NetFlow** solves this:

- Only 1 in N packets is examined (N is configurable — 1:100 to 1:10000)
- Byte and packet counts are extrapolated (multiplied by N)
- Results are approximate — acceptable for capacity planning, not for forensics

For security investigations, sampled NetFlow still detects patterns (port scans, beaconing, large transfers) even if exact byte counts are off.

## Cisco Configuration (IOS / IOS-XE)

### Classic NetFlow v5 (IOS 12.x)

```
! Enable on each interface (ingress and egress separately)
interface GigabitEthernet0/0
 ip flow ingress
 ip flow egress

! Export to collector
ip flow-export version 5
ip flow-export destination 10.0.1.100 2055
ip flow-export source Loopback0

! Tune timeouts (optional)
ip flow-cache timeout inactive 15
ip flow-cache timeout active 1
```

### Flexible NetFlow (IOS-XE, recommended)

```
! Define what fields to capture
flow record MY-RECORD
 match ipv4 protocol
 match ipv4 source address
 match ipv4 destination address
 match transport source-port
 match transport destination-port
 match interface input
 collect counter bytes
 collect counter packets
 collect timestamp sys-uptime first
 collect timestamp sys-uptime last

! Define exporter
flow exporter MY-EXPORTER
 destination 10.0.1.100
 source Loopback0
 transport udp 2055
 export-protocol ipfix

! Bind to monitor
flow monitor MY-MONITOR
 record MY-RECORD
 exporter MY-EXPORTER
 cache timeout active 60
 cache timeout inactive 15

! Apply to interface
interface GigabitEthernet0/0
 ip flow monitor MY-MONITOR input
 ip flow monitor MY-MONITOR output
```

## Security Use Cases

NetFlow is one of the most effective passive security monitoring tools because it captures **all traffic** without needing to decrypt it.

**Port scan detection**: A single source making connections to many destination IPs on the same port, or sweeping many ports on one destination, creates a distinct flow pattern — high flow count, low byte count, many unique destinations/ports.

**Data exfiltration**: Large outbound transfers from an internal host to an unfamiliar external IP, especially outside business hours. `src=10.x.x.x dst=185.x.x.x bytes=40GB duration=2h`.

**Lateral movement**: Unusual internal traffic — a workstation talking to many servers on port 445 (SMB) or port 22 (SSH) in rapid succession. Normal workstations don't do this.

**C2 beaconing**: Periodic small flows from an internal host to the same external IP — e.g., 200 bytes every 30 seconds for days. Classic command-and-control pattern.

**DNS exfiltration**: Abnormally high DNS query volume from one host. Each DNS query is a separate UDP flow — a host making 10,000 queries/hour stands out.

## Capacity Planning Use Cases

- **Top talkers per interface**: identify which hosts consume the most WAN bandwidth
- **Application breakdown**: what percentage of traffic is backup, VoIP, web, database?
- **Peak time analysis**: when does utilisation spike? Plan maintenance windows around it.
- **Trend analysis**: is WAN utilisation growing 10% per month? Plan upgrades.
- **Backup job impact**: identify that nightly backup to S3 saturates the WAN link 01:00–04:00.
