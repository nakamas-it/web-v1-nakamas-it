## Overview

Proxmox Virtual Environment (Proxmox VE) is a complete, open-source virtualisation platform published by Proxmox Server Solutions GmbH in Vienna. It ships as a Debian Linux-based OS image: you boot the installer, and within minutes you have a fully operational hypervisor reachable via a web browser. No separate management server, no per-CPU licensing, no annual subscription required to access enterprise features — the Community Edition is genuinely free.

The platform unifies two complementary virtualisation technologies on a single host: **KVM** (Kernel-based Virtual Machine) for full hardware-isolated virtual machines, and **LXC** (Linux Containers) for lightweight OS-level containers. A Proxmox cluster can run both on the same nodes, managed through the same interface, with the same backup tooling, storage abstraction, and HA framework.

This combination — KVM for full isolation, LXC for efficiency, ZFS for local storage, Ceph for distributed storage, all free — is why Proxmox has become the dominant choice in homelabs, growing SMBs, and enterprise environments where VMware's post-Broadcom pricing is no longer acceptable.

## What Proxmox VE Is

Proxmox VE is a Linux distribution, not a product installed on top of one. The base is **Debian 12 (Bookworm)** at current versions. On top of Debian, Proxmox ships:

- A custom Linux kernel with the KVM module built in
- QEMU for device emulation
- LXC userspace tooling
- The `pve-manager` package — the web UI, API, and cluster management
- Corosync — distributed cluster membership and quorum
- DRBD/ZFS/Ceph integration packages

The host is a standard Linux system. You can SSH in, run `apt`, install packages, inspect running processes, examine kernel logs. This is both an advantage (full control, familiar tooling) and a consideration (the host is a general-purpose OS, not a purpose-hardened hypervisor micro-kernel).

All management is available through:
- **Web UI** — browser-based, served on port 8006 (HTTPS). Complete VM/container lifecycle, storage management, cluster views, console access.
- **`pvesh` / REST API** — full programmatic access. The web UI itself is a React frontend consuming this API.
- **`pvecm`, `qm`, `pct`, `pvesm`** — CLI tools for cluster management, KVM VMs, LXC containers, and storage respectively.
- **Proxmox VE API** — used by Terraform, Ansible modules, and third-party integrations.

## KVM — Kernel-based Virtual Machine

KVM is a Linux kernel module (`kvm.ko`, `kvm-intel.ko` or `kvm-amd.ko`) that transforms the Linux kernel into a Type 1 hypervisor. When KVM is loaded, the kernel gains the ability to host VMs with near-native CPU performance using Intel VT-x or AMD-V hardware extensions.

KVM alone is not a complete virtualisation solution — it provides the CPU and memory isolation primitives, but not device emulation. **QEMU** (Quick Emulator) provides the device model: virtual NICs, virtual disk controllers, virtual USB, display adapters. KVM handles the CPU; QEMU handles everything else.

In Proxmox, each KVM VM runs as a `qemu-system-x86_64` process on the host. That process is a normal Linux process — it has a PID, it appears in `ps`, it can be `strace`d. Proxmox wraps QEMU process management through `qm` commands and the web UI.

### VirtIO — Paravirtual Drivers

By default, QEMU can emulate real hardware devices that any OS can use without modification (e1000 NIC, IDE disk controller). This emulation is convenient but slow — every I/O operation from the guest triggers a VM exit, is processed in software by QEMU, and returned to the guest.

**VirtIO** is a Linux-standard paravirtualisation framework that eliminates this overhead. The guest loads a VirtIO driver that communicates directly with QEMU via a shared-memory ring buffer (virtqueue), without going through device emulation. VirtIO drivers are included in the Linux kernel by default (from 2.6.25 onward) and are available for Windows via the Red Hat VirtIO driver package.

Key VirtIO devices in Proxmox KVM VMs:

| Device | VirtIO driver | Replaces |
|---|---|---|
| Network | `virtio-net` | Emulated e1000, rtl8139 |
| Disk (block) | `virtio-blk` | Emulated IDE |
| Disk (SCSI) | `virtio-scsi` | Emulated LSI SCSI |
| Memory balloon | `virtio-balloon` | None — no emulated equivalent |
| Serial console | `virtio-serial` | Emulated serial port |
| RNG (entropy) | `virtio-rng` | None |

For production KVM VMs on Proxmox, always configure VirtIO network and storage. The performance difference vs emulated devices is substantial — particularly for disk I/O, where emulated SCSI can bottleneck at 50–100 MB/s while VirtIO-SCSI saturates storage at multiple GB/s.

### QEMU Machine Types

VMs have a **machine type** (`q35` or `i440fx`) that determines the virtual chipset presented to the guest:

- **i440fx** — Intel 440FX chipset. Legacy PCI bus. Required for some older OS compatibility.
- **q35** — Intel Q35 chipset. PCIe bus. Required for UEFI boot, NVMe emulation, and IOMMU-based PCI passthrough. Use for all modern guests.

The machine type is set at VM creation and changing it later is disruptive (requires VM shutdown). Use q35 for any VM that may need PCI passthrough or UEFI.

## LXC — Linux Containers

LXC (Linux Containers) is OS-level virtualisation. Unlike KVM, which provides a complete virtual machine with its own kernel, LXC containers share the Proxmox host's kernel. Isolation is provided by two Linux kernel features:

- **Namespaces** — each container has its own isolated view of process IDs, network interfaces, mount points, hostname, users, and IPC. A process inside a container can only see other processes in the same container's PID namespace.
- **cgroups (control groups)** — limit and account for resource consumption. A container's CPU, RAM, I/O, and network bandwidth are bounded by cgroup policies enforced by the kernel.

The result is a container that looks and feels like an independent Linux system — its own IP address, its own filesystem, its own init process — but without the overhead of a separate kernel and virtual hardware. LXC containers start in under a second and consume tens of megabytes of overhead versus hundreds of megabytes for a full KVM VM.

### What LXC Is Good For

LXC excels for Linux services that do not require kernel modification:

- Web servers (nginx, Apache)
- Databases (PostgreSQL, MySQL, Redis)
- DNS servers (BIND, Unbound)
- Application servers (Java, Python, Node.js)
- Monitoring agents (Prometheus exporters, Telegraf)
- Internal tooling (GitLab, Nextcloud, Gitea)

### What LXC Cannot Do

- **Windows** — LXC shares the Linux kernel. Windows guests require KVM.
- **Kernel modules or custom kernels** — a container cannot load a kernel module; it uses the host's kernel. Applications requiring specific out-of-tree kernel modules (some VPN clients, certain hardware drivers) may not work in LXC.
- **Docker inside LXC** — possible with a privileged container or specific nesting configuration, but adds complexity. The typical recommendation is to run Docker inside a KVM VM if full isolation is important.
- **Privilege-sensitive operations** — operations that require root access to the real host kernel (raw socket manipulation at a level that bypasses the network namespace, `/dev/mem` access) will fail in an unprivileged container.

Proxmox ships a library of **LXC templates** — minimal Debian, Ubuntu, Alpine, CentOS, Fedora, and other images that boot in seconds. These are not Docker images; they are complete root filesystems intended to run a full init system (systemd or OpenRC) as PID 1.

## Storage Backends

Proxmox abstracts storage through a unified storage model. Each storage is defined in `/etc/pve/storage.cfg` and appears as a named backend in the UI. VM disks and container rootfs volumes can be placed on any configured storage.

### Local Storage

| Type | Description | Snapshot support | Notes |
|---|---|---|---|
| `dir` (directory) | Files on a mounted filesystem (ext4, XFS) | Qcow2 only | Simple; no block-level snapshots |
| LVM | Logical volumes from an LVM volume group | No | Fastest for block I/O; no COW snapshots |
| LVM-thin | Thin-provisioned LVM | Yes (LVM thin snapshots) | Snapshot support; slight overhead vs thick LVM |
| ZFS | ZFS datasets or zvols | Yes (ZFS snapshots) | Preferred for local storage — see below |

### Network Storage

| Type | Description |
|---|---|
| NFS | NAS share mounted and used as a `dir` backend |
| CIFS/SMB | Windows/Samba share |
| iSCSI | Block storage presented as iSCSI LUN; formatted with LVM or used as raw |
| Ceph RBD | Distributed Ceph block devices — see Ceph section |

## ZFS on Proxmox

ZFS is the preferred local storage backend for Proxmox and is deeply integrated into the platform. Proxmox includes ZFS support in the base installation — no separate packages required.

### Why ZFS

ZFS is not just a filesystem. It is a combined volume manager and filesystem with capabilities that directly benefit virtualisation:

- **Copy-on-Write (COW)** — writes never overwrite existing data. A new block is written, then the metadata is updated atomically. Power failure mid-write leaves the previous consistent state intact.
- **Snapshots** — instantaneous, space-efficient. A snapshot consumes zero space at creation; space accumulates only as the parent dataset diverges. Snapshots are taken at the ZFS level, independent of the guest OS.
- **Send/receive** — ZFS can stream snapshots to another ZFS pool locally or over the network. This is the foundation for Proxmox's built-in ZFS replication.
- **Data integrity** — every block has a checksum stored in the metadata tree. On read, the checksum is verified; silent data corruption is detected and, with redundant pools, automatically corrected.
- **Compression** — transparent compression (LZ4 by default) reduces actual storage consumed. LZ4 compression is fast enough that it often *improves* I/O throughput by reducing the amount of data written to disk.
- **Deduplication** — block-level deduplication (RAM-intensive; typically not recommended unless very large ARC and specific workloads).

### ZFS Pool Configurations

| Configuration | Fault tolerance | Minimum disks | Notes |
|---|---|---|---|
| **Stripe (RAID-0)** | None | 1 | No fault tolerance; full capacity |
| **Mirror (RAID-1)** | 1+ disks per mirror | 2 | N-way mirrors possible (3-way for extra protection) |
| **RAID-Z1** | 1 disk | 3 | RAID-5 equivalent; avoid for large disks (long resilver) |
| **RAID-Z2** | 2 disks | 4 | RAID-6 equivalent; recommended for capacity-heavy pools |
| **RAID-Z3** | 3 disks | 5 | Maximum parity; for very large disk arrays |

For a Proxmox homelab or SMB deployment with 2 NVMe drives: **mirror** is the correct choice. Two-way mirror provides instant resilver, no RAID-Z read performance overhead, and tolerates one disk failure.

### ZFS Caching Architecture

ZFS uses a multi-tier caching system that is important to understand for Proxmox deployments:

- **ARC (Adaptive Replacement Cache)** — ZFS's primary read cache, stored in system RAM. By default, ZFS claims up to half of system RAM for ARC. This is intentional — the more RAM, the better the cache hit rate. For a Proxmox host with 128 GB RAM, ZFS may use 40–60 GB as ARC. This is not a memory leak; it is cache and will be released under memory pressure.
- **L2ARC (Level 2 ARC)** — an optional SSD used as a secondary read cache, extending ARC capacity beyond available RAM. Useful when read I/O patterns exceed ARC but fit in SSD. Requires a dedicated fast SSD (NVMe preferred). L2ARC metadata is stored in RAM — each GB of L2ARC consumes roughly 10 MB of RAM for metadata.
- **SLOG (Separate Log device / ZIL)** — ZFS uses a **ZFS Intent Log (ZIL)** to acknowledge synchronous writes immediately while destaging to the main pool asynchronously. By default, the ZIL lives on the main pool, which serialises sync writes. A dedicated SLOG device (a fast NVMe or Optane SSD) places the ZIL on a separate, low-latency device. This dramatically improves sync write performance for workloads that issue fsync() — NFS servers, databases, mail servers, any application that calls O_SYNC. For Proxmox NFS datastores or VMs running PostgreSQL/MySQL, a SLOG is a significant performance multiplier.

**ECC RAM**: ZFS's data integrity guarantees depend on the data being correct when it reaches ZFS. If RAM is faulty and silently flips bits in the write path, ZFS will faithfully write corrupt data and store a valid checksum for that corrupt data — the checksum will pass on read, and the corruption is undetected. ECC RAM eliminates this risk. It is the standard recommendation for any ZFS production deployment, though many homelabs run ZFS without ECC successfully.

## Proxmox Clustering

A Proxmox cluster groups multiple Proxmox nodes under a single management pane. Up to 32 nodes per cluster. Cluster membership and quorum are managed by **Corosync**, the same distributed messaging layer used in Linux Pacemaker clusters.

### Cluster Communication

Corosync communicates over a dedicated cluster network (separate from VM traffic) using UDP multicast or unicast. Each node sends heartbeats; the cluster tracks which nodes are reachable. Corosync uses a **quorum** model — a majority of nodes must be reachable for the cluster to operate. A 3-node cluster needs 2 nodes; a 5-node cluster needs 3.

For 2-node clusters, a **QDevice (Quorum Device)** — a small third node running `corosync-qnetd` — provides the tiebreaker vote without requiring a full Proxmox node. This enables 2+1 vote distribution, preventing a 2-node cluster from losing quorum on a single node failure.

Unlike VMware's cluster model, Proxmox clusters do **not** require shared storage. Each node can have its own local storage. VMs can be migrated only if the target node has access to the source storage (either shared storage or local storage with replicated data). This is more flexible but requires explicit planning for where VM data lives.

### Cluster Features

- **Unified web UI** — manage all nodes, their VMs, and their storage from any node's web interface.
- **Single configuration database** — VM configurations, storage definitions, and user accounts are stored in `/etc/pve/`, which is a Corosync-replicated POSIX filesystem backed by pmxcfs (Proxmox cluster filesystem). Changes on one node propagate to all nodes automatically.
- **Centrally managed SDN** — Software Defined Networking allows VLAN configurations and network overlays (VXLAN, EVPN) to be managed cluster-wide.
- **Distributed firewall** — per-VM, per-container, and cluster-level firewall rules managed centrally and applied on each node.

## High Availability (HA)

Proxmox HA provides automatic VM and container restart on node failure. It is a cluster feature — HA requires at least 3 nodes to maintain quorum after losing one.

### How Proxmox HA Works

HA is managed by the `pve-ha-manager` service. Two roles:

- **HA Manager (LRM — Local Resource Manager)** — runs on every node. Manages the resources (VMs/containers) assigned to HA on that node. Reports resource state to the cluster manager.
- **HA Manager (CRM — Cluster Resource Manager)** — elected master coordinates the HA state machine across the cluster.

**Fencing is mandatory for correct HA operation.** Before Proxmox restarts VMs from a failed node, it must ensure the failed node is actually powered off. Without fencing, a node that is merely network-isolated (reachable by power but not the cluster) might still have its VMs running when the cluster restarts them on another node. Two nodes writing to the same VHDX simultaneously corrupts it. Fencing eliminates this risk by cutting power to the node before proceeding.

Supported fencing mechanisms: IPMI (most common), iDRAC, iLO, APC PDU, custom scripts. In a homelab without IPMI, the `watchdog` hardware watchdog timer can serve as a fencing mechanism — the failing node resets itself when it loses quorum.

## Ceph Integration

Proxmox ships with native Ceph integration — not as an external storage array, but as a distributed storage system built from the same Proxmox nodes. Each node in a Ceph cluster contributes its local disks as **OSDs (Object Storage Daemons)**, and Ceph distributes data across all OSDs with configurable redundancy.

### Ceph Architecture (in Proxmox context)

| Component | Role |
|---|---|
| OSD (Object Storage Daemon) | One per disk. Stores objects, handles replication, participates in CRUSH placement. |
| MON (Monitor) | Maintains the cluster map (CRUSH map, OSD map). Quorum-based; minimum 3. |
| MGR (Manager) | Metrics, dashboard, orchestration module. Runs alongside MON. |
| MDS (Metadata Server) | Required only for CephFS (Ceph filesystem). Not needed for RBD block storage. |

Proxmox manages all of this from the web UI — adding OSDs, creating MONs, and creating RBD pools is point-and-click. The Ceph pool is then added as a Proxmox storage backend, and VM disks are created as RBD images in that pool.

### Ceph Data Distribution — CRUSH

Ceph places data using the **CRUSH (Controlled Replication Under Scalable Hashing)** algorithm. Rather than a lookup table, CRUSH uses a deterministic algorithm: given a data object's name and the current cluster map, any client (or OSD) can independently compute where each replica of that object lives. No lookup server needed.

The default replication policy places each object's N replicas on N different hosts (not just different OSDs). A 3-way replicated object exists on 3 separate Proxmox nodes. If one node fails, the other two nodes have the complete data set; Ceph re-replicates to restore the replica count.

**Minimum Ceph cluster for production**: 3 nodes. With 2 nodes, a single node failure removes quorum and the cluster stops serving I/O entirely.

### Ceph vs ZFS for Proxmox Storage

| Property | Ceph RBD | ZFS (local) |
|---|---|---|
| Minimum nodes | 3 | 1 |
| I/O latency | Higher (network round-trips) | Lower (local NVMe) |
| Throughput scaling | Scales with cluster size | Limited to local NVMe speed |
| Fault tolerance | Survives node failure; data still on other nodes | Survives disk failure; node failure = data unavailable |
| Live migration | Any node to any node (data is everywhere) | Requires shared or replicated storage |
| Complexity | High (MON quorum, OSD management, network) | Low (local ZFS pool) |
| Best for | Clusters with 3+ nodes needing shared storage | Single nodes or 2-node clusters |

A common Proxmox architecture for a 3-node cluster: ZFS mirrors on each node for fast local VM storage, plus Ceph RBD for VMs that need live migration between any node. The two storage backends coexist in the same cluster.

## Backup — vzdump and Proxmox Backup Server

Proxmox includes `vzdump`, a built-in backup tool that creates consistent backups of running VMs and containers. For KVM VMs, vzdump uses QEMU's live backup capability with change-block tracking. For LXC containers, it suspends the container briefly (or not at all in snapshot mode) and archives the container filesystem.

Backup destinations can be local directories, NFS mounts, or a **Proxmox Backup Server (PBS)**.

### Proxmox Backup Server

PBS is a dedicated backup server application — also Debian-based, also free — that stores Proxmox backups with **incremental, chunk-level deduplication**. After the initial full backup, subsequent backups transfer only changed data chunks. A 64 GB VM with moderate daily change rate might produce a full backup of 20 GB compressed, and daily incrementals of 1–3 GB.

PBS stores backups in **datastores** with configurable retention policies (keep last N daily, N weekly, N monthly) enforced automatically during garbage collection.

The efficiency gain over ESXi's native backup (which takes a full snapshot per backup) is significant. PBS is the correct backup solution for any production Proxmox deployment.

## Web UI Overview

The Proxmox web UI is comprehensive enough that SSH is rarely needed for day-to-day operations:

- **Datacenter level**: cluster summary, HA configuration, storage definitions, SDN, firewall rules, user management, backups, Ceph management.
- **Node level**: per-host CPU/RAM/network graphs, shell access, disk management, ZFS pool creation, network interface configuration, package updates.
- **VM/Container level**: hardware configuration, snapshot management, backup, console (VNC, SPICE, or xterm.js), live migration, clone.

The console access is particularly clean: xterm.js provides a full browser-based terminal for both the host and containers, SPICE provides a high-performance VM console for VMs with graphical desktops.

## Common Scenarios

**Homelab single node with ZFS**: One server, 2× NVMe in ZFS mirror. 4 KVM VMs (pfSense router, Ubuntu server, Windows development machine, Kali Linux) and 8 LXC containers (Pi-hole, Nextcloud, Gitea, Prometheus/Grafana stack, Nginx Proxy Manager). PBS running in an LXC container for backup. Total overhead for the 8 containers: under 2 GB RAM.

**3-node SMB cluster**: Three identical servers, each with 2× NVMe mirror (ZFS) and 4× SSD for Ceph OSDs. Corosync cluster with 3 MONs. HA enabled for critical VMs. Live migration between any two nodes via Ceph RBD storage. PBS on a fourth small node for backups.

**Migration from ESXi after Broadcom licensing change**: ESXi VMs exported as OVA. Proxmox imports OVA directly via `qm importovf`. VMDK disks converted to QCOW2 or raw on Proxmox storage. VMware Tools removed, VirtIO drivers installed (for Linux, kernel drivers already present; for Windows, Red Hat VirtIO package installed). Migration of a 20-host VMware cluster to 5 more powerful Proxmox nodes, saving substantial licensing costs.

## Proxmox vs ESXi Comparison

| Property | Proxmox VE | VMware ESXi + vCenter |
|---|---|---|
| Licensing | Free (Community Edition); optional paid support | Per-CPU subscription (Broadcom) |
| Hypervisor | KVM (Linux kernel) | VMkernel (proprietary) |
| Container support | Native LXC | None natively (separate NSX-T/Tanzu) |
| Management plane | Included (pve-manager) | vCenter Server (separate product/licence) |
| Live migration | KVM live migration (shared or Ceph) | vMotion (shared or vSAN) |
| HA | Built-in (Corosync + pve-ha-manager) | vSphere HA (vCenter required) |
| Hyper-converged storage | Ceph (built-in) | vSAN (additional licence) |
| Backup | vzdump + Proxmox Backup Server (free) | Requires third-party (Veeam, etc.) |
| Local storage | ZFS (built-in, excellent) | VMFS (no ZFS) |
| Support model | Community forum; paid Proxmox support tiers | VMware/Broadcom support contracts |
| Ecosystem | Open source, Ansible, Terraform provider | VMware ecosystem (PowerCLI, Aria, etc.) |
| Attack surface | Full Debian Linux | Minimal VMkernel |
| Best for | Homelabs, SMBs, cost-sensitive enterprise | Large enterprise, VMware-invested shops |
