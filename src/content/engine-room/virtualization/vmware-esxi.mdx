## Overview

VMware vSphere is not a single product — it is a stack. At the bottom is **ESXi**, the bare-metal hypervisor that runs directly on hardware and hosts virtual machines. Above it sits **vCenter Server**, the management plane that aggregates multiple ESXi hosts into clusters and enables the advanced features that make vSphere the dominant platform in enterprise data centres: live migration, high availability, automated load balancing, and hyper-converged storage.

Understanding what each component does, how they interact, and what actually happens under the hood when a VM migrates or a host fails is essential for anyone operating virtualised infrastructure. vSphere's abstractions are clean, but the mechanisms behind them are specific — and knowing those mechanisms is the difference between a reliable deployment and one that fails in ways you cannot explain.

---

## ESXi — The Hypervisor

ESXi is a **Type 1 bare-metal hypervisor**. It does not run on top of a general-purpose OS — it boots directly from firmware, loads its own micro-kernel called **VMkernel**, and takes exclusive ownership of all hardware resources on the host.

VMkernel is purpose-built. It provides only what a hypervisor needs: CPU scheduling, memory management, a storage I/O stack, and a network stack. There is no shell in the traditional sense. Remote management happens through:

- **DCUI (Direct Console User Interface)** — the keyboard/monitor console available at the physical machine. Used for initial IP configuration and emergency access.
- **SSH** — enabled manually (disabled by default per VMware's security hardening guidance). A BusyBox-based shell with a subset of Unix commands plus VMware-specific utilities (`esxcli`, `vim-cmd`, `esxtop`).
- **vSphere Client (web)** — browser-based UI served by ESXi itself over port 443. Primary management interface when vCenter is unavailable.
- **vCenter API (REST/SOAP)** — the primary programmatic interface for automation.

### VM Files

Each virtual machine is a collection of files stored on a datastore. The critical files:

| File extension | Purpose |
|---|---|
| `.vmx` | VM configuration — vCPU count, RAM, hardware version, network adapters, disk references |
| `.vmdk` | Virtual disk descriptor (text file) — points to the flat file |
| `-flat.vmdk` | The actual virtual disk data — the raw bytes of the guest's storage |
| `.nvram` | VM firmware state — BIOS/UEFI settings, boot order |
| `.vmem` | Guest RAM swapped to disk when host memory is overcommitted |
| `.vswp` | VM swap file — pre-allocated at VM power-on |
| `.log` | VMware log file — per-VM event and error log |
| `.vmsd` | Snapshot metadata |
| `.delta.vmdk` | Snapshot redo log — writes after snapshot creation go here |

### Datastores and VMFS

ESXi stores VM files on **datastores** — formatted storage volumes presented to the host. The native filesystem for block storage is **VMFS (Virtual Machine File System)**, a cluster-aware filesystem specifically designed for shared storage access from multiple hosts simultaneously.

VMFS uses file-level locking on individual `.vmdk` files, allowing multiple ESXi hosts to read from the same datastore concurrently while ensuring only one host writes to any given VM's disk at a time. This shared-access property is what makes cluster features like vMotion and HA possible — VMs are visible to all hosts in the cluster.

Supported datastore types:

- **VMFS on SAN/iSCSI** — the traditional enterprise model. Block storage from a Fibre Channel or iSCSI SAN, formatted as VMFS.
- **NFS** — NAS storage mounted directly. No VMFS layer — NFS handles concurrency. Simpler, slightly higher latency for write-intensive workloads.
- **vSAN** — hyperconverged storage built from local disks in each ESXi host. Covered below.
- **vVols (Virtual Volumes)** — per-VM storage objects directly managed by a storage array. Bypasses VMFS; the array handles per-VM snapshots and policies natively.

## vCenter Server — The Management Plane

A standalone ESXi host manages itself. The moment you need two ESXi hosts to cooperate — for live migration, HA, or shared load balancing — you need vCenter.

**vCenter Server** is the centralised management platform that aggregates ESXi hosts, configures clusters, and provides the single interface through which all vSphere infrastructure is operated. Without vCenter:

- You cannot live-migrate VMs between hosts (vMotion requires vCenter to orchestrate the transfer)
- HA does not exist — each host is independent
- DRS cannot rebalance VMs — there is nothing to compare host utilisation against
- Distributed switches cannot be configured — each host manages its own switching independently

**vCenter Server Appliance (VCSA)** is the current deployment form. It is a Linux VM running VMware's Photon OS, pre-packaged as an OVA. It contains the vCenter service, an embedded PostgreSQL database (for environments up to ~2,000 hosts / ~35,000 VMs), and the vSphere web client. Deploying VCSA to the same cluster it manages introduces a bootstrapping consideration: if the cluster loses power, vCenter must be started before HA can restart other VMs — it cannot restart itself.

vCenter organises infrastructure through a logical hierarchy: **Datacenter → Cluster → Host → VM**. All configuration, permissions, and policies are applied at the appropriate level and inherited downward.

### vCenter Network Ports (Key)

| Port | Protocol | Purpose |
|---|---|---|
| 443 | HTTPS | vSphere Client, REST API, SDK |
| 902 | TCP/UDP | ESXi host-to-vCenter heartbeat; legacy vSphere Client |
| 9443 | HTTPS | vSphere Web Client (legacy, pre-6.5) |
| 5480 | HTTPS | VCSA management console (appliance configuration) |
| 636 | LDAPS | Identity source (Active Directory) |

ESXi hosts communicate with vCenter over port 443. vMotion traffic moves on a dedicated VMkernel port group (typically port 8000 between hosts, but this is negotiated per-migration).

## vSphere Clustering

A **vSphere Cluster** is a group of ESXi hosts managed as a unit by vCenter. Hosts in a cluster share:

- A resource pool (total CPU and RAM visible to DRS)
- Shared storage (VMFS datastore or vSAN) — required for vMotion and HA
- Cluster-level HA and DRS policies

VMs running on a clustered host are available to be migrated between all hosts in the cluster. This shared operational model is what transforms a collection of individual servers into a resilient, self-managing infrastructure.

## vMotion — Live VM Migration

vMotion is VMware's implementation of live migration: moving a running VM from one ESXi host to another with no guest downtime. From the VM's perspective, nothing happens. TCP connections stay established, application state is preserved, memory is intact.

**Requirements for vMotion:**

- **Shared storage** — both hosts must be able to access the VM's datastore. The VM's `.vmdk` is not moved during vMotion (that is Storage vMotion). Only RAM and CPU state are transferred.
- **Compatible CPU features** — the source and destination CPUs must expose the same instruction set to the guest. A VM cannot vMotion from an Intel Skylake host to a Broadwell host if the guest has Skylake-specific instructions in its running state.
- **EVC (Enhanced vMotion Compatibility)** — a cluster-level setting that masks the CPU feature set presented to all VMs to the lowest common denominator in the cluster. Enables vMotion between hosts with different CPU generations at the cost of not exposing newer CPU features to VMs.
- **Dedicated vMotion network** — vMotion traffic (typically 10 Gb/s) requires a dedicated VMkernel port on a high-bandwidth network. The duration of the final stop-and-copy phase is inversely proportional to network bandwidth.

**Storage vMotion** is the complementary operation: moving a VM's virtual disk between datastores while the VM runs. The disk contents are mirrored to the new datastore in the background, with a change-block tracking (CBT) mechanism ensuring writes during migration are captured. Useful for moving VMs off an ageing storage array without downtime.

**vMotion across datacentres** (Long-Distance vMotion) is possible over high-latency WAN links with some constraints on round-trip time. Metro-distance migrations (< 5 ms RTT) are routine; cross-continental is technically possible but uncommon.

## High Availability (HA)

vSphere HA provides automatic restart of VMs when an ESXi host fails. It is not live migration — there is a restart period during which the VM's processes are down. HA is the difference between "someone gets paged at 3 AM to manually restart 40 VMs" and "the cluster handles it in under 3 minutes without human intervention."

### How HA Works

vCenter configures a HA cluster by installing HA agents on each host. One host is elected the **primary** via a distributed election; the others are **secondary** agents. The primary monitors all hosts in the cluster.

**Failure detection** uses two independent heartbeat mechanisms:

1. **Network heartbeats** — HA agents exchange heartbeats every second over the management network. If the primary does not receive heartbeats from a host for 10 seconds (default), it suspects failure.
2. **Datastore heartbeats** — ESXi hosts write a heartbeat to a designated datastore every 2 seconds. If network heartbeats fail but datastore heartbeats continue, the host is considered **isolated** (network partition), not failed. An isolated host stops running its VMs only if configured to do so — this avoids split-brain VM restarts.

When a host is declared failed:
1. The primary queries the datastore heartbeat file one more time to confirm the host is genuinely down (not just network-isolated).
2. The primary identifies which VMs were running on the failed host.
3. HA restarts those VMs on surviving hosts with sufficient capacity.

### Admission Control

HA requires that the cluster always hold sufficient capacity to absorb failures. **Admission control** enforces this:

- **Percentage-based**: Reserve N% of cluster CPU and RAM at all times. Example: with 25% reserved on a 4-host cluster, the cluster can survive one host failure. Powering on a new VM that would consume reserved capacity is blocked.
- **Host failures to tolerate**: Reserve enough capacity to restart all VMs from N hosts. VMware calculates the required reservation based on the largest N hosts in the cluster.

Without admission control, a cluster could be 100% utilised and HA would be incapable of restarting failed VMs — defeating its purpose.

### HA vs Fault Tolerance

| Property | HA | Fault Tolerance (FT) |
|---|---|---|
| Mechanism | VM restart after host failure | Shadow VM runs in lockstep on second host |
| Downtime on failure | Seconds to minutes (restart) | Zero — shadow takes over instantly |
| vCPU limit | No limit | 4 vCPUs max |
| RAM | No limit | No formal limit; network bandwidth dependent |
| Performance overhead | None until failure | ~10% (continuous state synchronisation) |
| Storage requirement | Shared storage | Shared storage |
| Use case | General VMs | Extremely critical, low-vCPU workloads |

FT uses the **vLockstep protocol**: every CPU instruction the primary VM executes is replicated to the shadow VM on a secondary host in real time. If the primary host fails, the shadow takes over instantly — same memory state, same TCP connections, zero disruption. The bandwidth required to keep two VMs in lockstep limits practical use to low-CPU-intensity workloads.

## Distributed Resource Scheduler (DRS)

DRS monitors CPU and RAM utilisation across all hosts in a cluster and uses vMotion to rebalance VMs when load becomes uneven. Without DRS, VMs stay on whatever host they powered on — a host that gets heavily loaded stays heavily loaded while adjacent hosts sit idle.

DRS operates on a continuous basis: every 5 minutes, it evaluates the load across all hosts. If the standard deviation of host utilisation exceeds a configurable threshold, DRS recommends (or automatically executes) vMotion operations to bring utilisation back into balance.

**DRS automation levels:**

- **Fully Automated** — DRS migrates VMs automatically without prompting. Most common in production.
- **Partially Automated** — DRS places VMs on initial power-on automatically but requires human approval for ongoing migrations.
- **Manual** — DRS presents recommendations only. Administrator approves each vMotion.

**DRS rules** provide explicit controls to override DRS decisions:
- **VM-VM affinity rules** — force specific VMs to always run on the same host (e.g., tightly coupled application tiers) or always on different hosts (e.g., primary and secondary database instances).
- **VM-Host affinity rules** — require VMs to run on specific hosts or groups of hosts (e.g., licenced software bound to specific hardware).

DRS is frequently misunderstood as load balancing in the traditional sense — it is not reacting to instantaneous spikes. It balances on a 5-minute cycle, optimising for sustained utilisation rather than instantaneous peaks.

## vSAN — Hyperconverged Storage

vSAN eliminates the need for an external SAN or NAS by building a shared datastore directly from local NVMe, SSD, and HDD devices in each ESXi host. It is VMware's hyperconverged storage solution.

### Architecture

Each ESXi host that participates in vSAN contributes its local storage to a **vSAN datastore** — a single logical storage pool visible to all hosts in the cluster. There is no external storage controller; the storage controller is distributed across all hosts.

vSAN organises storage into **disk groups** on each host. In the current architecture (Express Storage Architecture, vSAN 7.0+), each host has a single pool of NVMe/SSD devices. In the older hybrid architecture, each disk group has one SSD (as a write cache and read cache) and one or more HDDs (as capacity).

### Data Placement — Storage Policies

vSAN is **policy-driven**: you define a VM Storage Policy and assign it to a VM or individual VMDK. vSAN enforces the policy by choosing how to distribute the data across hosts. The key policy components:

- **Failures To Tolerate (FTT)**: How many simultaneous host/disk/network failures the data must survive. FTT=1 requires data to exist on a minimum of 3 hosts. FTT=2 requires 5 hosts.
- **RAID method**:
  - **RAID-1 (mirroring)** — full copies on N+1 hosts. Space overhead = FTT+1 × object size. Simple, reliable.
  - **RAID-5/6 (erasure coding)** — data striped with parity across hosts. More efficient (FTT=1 RAID-5 uses 1.33× space vs 2× for RAID-1 mirroring) but requires minimum 4 hosts (RAID-5) or 6 hosts (RAID-6).

A VM with an FTT=1 RAID-1 policy has its VMDK stored as two complete copies on two different hosts, plus a **witness** component (metadata only, no data) on a third host. If one data host fails, the surviving copy plus the witness maintain quorum and the VM keeps running. vSAN then rebuilds the failed component when the host returns or is replaced.

### vSAN Stretched Cluster

vSAN supports a stretched cluster topology across two physical sites (e.g., two data centre buildings) with a witness VM at a third site. VMs have data at both sites simultaneously. If site A fails, site B has a complete copy and the witness provides quorum — VMs on site A are restarted at site B. This provides active-active storage with site-level failure tolerance.

## Distributed Switch (vDS)

Each ESXi host has one or more **virtual switches** that connect VM virtual NICs to the physical network uplinks. A standard vSwitch is configured host-by-host — port group settings on one host are independent of identical-looking port groups on another host.

The **vSphere Distributed Switch (vDS)** is managed at the vCenter level and spans all hosts in the cluster. A port group configured on the vDS is automatically consistent across every host — same VLAN, same traffic shaping policy, same security settings, same NIC teaming. From the network's perspective, the vDS presents a single logical switch spanning all physical hosts.

This matters operationally: without vDS, a vMotion that moves a VM from Host A to Host B succeeds only if both hosts have an identically configured port group. With vDS, consistency is guaranteed by construction.

**NIOC (Network I/O Control)** on a vDS allows per-traffic-type bandwidth reservations: vMotion traffic gets a minimum guaranteed bandwidth, HA heartbeat traffic is protected from being starved by bulk backup traffic, and so on.

## VMware Tools

VMware Tools is a set of drivers and utilities installed inside the guest OS. Without VMware Tools, VMs function but at degraded performance and with limited management integration.

**Performance drivers:**
- **VMXNET3** — paravirtualised NIC. Significantly higher throughput and lower CPU overhead than the emulated e1000 adapter. Standard for all new VMs.
- **PVSCSI** — paravirtualised SCSI controller. Higher IOPS and lower latency than the emulated LSI controller. Essential for I/O-intensive workloads.
- **Memory balloon driver** — allows the hypervisor to reclaim guest memory under pressure without swapping.
- **VSOCK** — efficient host-guest communication channel used by some advanced features.

**Management integration:**
- **Heartbeat** — vCenter knows the VM's guest OS is alive (not just that the VM process is running)
- **Guest shutdown/restart** — vCenter can cleanly shut down the guest OS rather than forcing a power-off
- **Quiesced snapshots** — VMware Tools coordinates with the guest OS VSS (Windows) or sync drivers (Linux) to quiesce the filesystem and application state before a snapshot is taken, ensuring the snapshot is application-consistent
- **Time synchronisation** — guest clock kept in sync with the hypervisor
- **Enhanced session mode (Windows)** — clipboard, audio, USB passthrough via VMRC console

Any VM running production workloads should have current VMware Tools installed. Running outdated Tools or no Tools is the single most common cause of unexplained VM performance problems reported as infrastructure issues.

## Licensing

VMware vSphere licensing (post-Broadcom acquisition in 2023) has shifted to a subscription model. Historically, the key tiers were:

| Feature | vSphere Standard | vSphere Enterprise Plus |
|---|---|---|
| ESXi hypervisor | Yes | Yes |
| vMotion | Yes | Yes |
| vSphere HA | Yes | Yes |
| DRS | No | Yes |
| vDS | No | Yes (required for vDS) |
| Fault Tolerance | No | Yes |
| Storage vMotion | No | Yes |
| NIOC | No | Yes |

In practical terms: **Standard** covers basic clustering and HA. **Enterprise Plus** is required for the advanced automation features (DRS, Storage vMotion, vDS) that make vSphere operationally efficient at scale.

vSAN is licensed separately per host and comes in different editions depending on the features required (compression, deduplication, stretched cluster).

## Common Scenarios

**Planned host maintenance**: Place the host in maintenance mode via vCenter. DRS automatically vMotions all running VMs to other hosts (Fully Automated DRS). Once the host shows zero running VMs, it is safe to reboot. After maintenance, DRS rebalances load across the cluster.

**Host hardware failure**: ESXi host loses power unexpectedly. HA primary detects missing heartbeats after 10 seconds, confirms via datastore heartbeat that the host is genuinely down, identifies 12 VMs that were running, and restarts them on surviving hosts with available capacity. Total restart time: 1–3 minutes depending on VM boot time.

**Storage migration**: Ageing iSCSI SAN is being decommissioned. New all-flash array presented to the cluster as a new datastore. Storage vMotion moves each VM's VMDK to the new datastore online, one at a time, with no VM downtime. Decommission old array once all VMs have migrated.

**Cross-version vMotion blocked**: Cluster has a mix of Intel Skylake and Cascade Lake hosts. EVC mode is not enabled. A VM running on a Cascade Lake host cannot vMotion to a Skylake host because the Cascade Lake CPU has exposed AVX-512 instructions to the guest that the Skylake host cannot provide. Enable EVC at the Intel Skylake baseline — VMs will no longer see AVX-512, but all vMotion operations will succeed.
