## TrueNAS CORE vs SCALE

TrueNAS is maintained by iXsystems and is the successor to FreeNAS. There are two active editions:

| Feature | TrueNAS CORE | TrueNAS SCALE |
|---------|-------------|---------------|
| Base OS | FreeBSD | Debian Linux |
| ZFS implementation | OpenZFS (FreeBSD) | OpenZFS (Linux) |
| Virtualisation | bhyve (FreeBSD hypervisor) | KVM |
| App hosting | FreeBSD jails | Kubernetes (k3s) + Docker |
| Clustering | No | Yes (Scale-Out TrueNAS) |
| GPU passthrough | Limited | Better support |
| Maturity | Very mature, conservative | Newer, actively developed |
| Recommended for | Maximum stability, pure NAS | App hosting, clustering, Linux ecosystem |

Both editions run ZFS and are functionally equivalent for NAS workloads (SMB, NFS, iSCSI). The choice is primarily about the additional capabilities on top of ZFS.

## ZFS Architecture

ZFS is not a traditional filesystem bolted onto a volume manager. It is an integrated filesystem, volume manager, and storage virtualisation layer. Understanding the layer stack is essential for correct TrueNAS configuration.

### Virtual Devices (vdevs)

A vdev is the fundamental redundancy unit in ZFS. All I/O for a pool goes through vdevs, and each vdev type provides a specific redundancy characteristic:

| vdev Type | Description | Faults Tolerated |
|-----------|-------------|-----------------|
| Single disk | No redundancy | 0 |
| Mirror | 2+ identical copies | N-1 disks in mirror |
| RAID-Z1 | Single parity (like RAID 5) | 1 disk |
| RAID-Z2 | Dual parity (like RAID 6) | 2 disks |
| RAID-Z3 | Triple parity | 3 disks |
| Special vdev | Metadata + small blocks | Mirrors only |
| Cache (L2ARC) | Read cache on SSD | N/A (loss = degraded perf) |
| Log (SLOG/ZIL) | Sync write journal on SSD | Mirror recommended |
| Spare | Hot spare disk | N/A |

**Critical constraint**: you cannot add more disks to an existing RAID-Z vdev. You can only add new vdevs to a pool. (TrueNAS SCALE 24.10+ introduced RAID-Z expansion, but it is slow and not yet general practice.) Plan vdev layout carefully before creating a pool.

### Storage Pools

A pool is composed of one or more vdevs. ZFS stripes data across all vdevs in a pool. If a pool has two RAID-Z1 vdevs of 4 disks each, data is striped across both vdevs, and each vdev can independently tolerate one disk failure.

Pool capacity = sum of usable capacity across all vdevs. A pool's read/write throughput scales with the number of vdevs.

### Datasets

Datasets are filesystem namespaces within a pool. Each dataset inherits pool properties but can override them independently:

- Compression algorithm and level
- Deduplication (on/off)
- Quota and reservation
- Access Control Lists (NFSv4 ACLs or POSIX ACLs)
- Snapshot policy
- Sync behaviour (standard, always, disabled)
- Record size (block size for the dataset)

Best practice: create separate datasets for different workloads — one for SMB shares, one for VM storage, one for backups. This allows independent snapshot policies, compression settings, and quotas.

### Zvols

A zvol is a block device created within a ZFS pool. It has a fixed provisioned size and presents as a raw block device. Zvols are used for:

- **iSCSI LUNs** — presented to ESXi, Hyper-V, or physical servers.
- **VM disk images** — as backing store for KVM/bhyve virtual machines.
- **Swap** — ZFS swap on FreeBSD.

Zvols have their own block size (volblocksize) — set this at creation time to match the workload (4K for databases, 64K–128K for sequential workloads).

## ZFS Data Integrity

ZFS was designed by Jeff Bonwick at Sun Microsystems with one non-negotiable principle: the filesystem must know when data is wrong, and it must fix it automatically when possible.

### End-to-End Checksumming

Every block of data written to a ZFS pool is checksummed (SHA-256 by default; fletcher4 for performance-sensitive workloads). The checksum is stored in the block's parent pointer — not alongside the block itself. This means a corrupted block cannot forge its own checksum.

On every read, ZFS recomputes the checksum and compares it to the stored value. If they diverge:

1. If the pool has no redundancy: the read fails with an error. Data is corrupted and unrecoverable from this copy.
2. If the pool has a mirror or RAID-Z: ZFS fetches the block from a redundant copy, verifies its checksum, returns the correct data to the application, and writes the corrected block back to the failed location. **Transparent, automatic repair.**

This catches a class of failure that RAID controllers cannot: silent data corruption (bit rot). A traditional RAID 5 controller checksums only the parity relationship — it cannot detect corruption on a single block that does not affect parity.

### Scrubbing

A scrub is a background read of every block in the pool to proactively verify checksums and repair any detected corruption while redundancy is still available.

```
# Run a scrub on pool named 'tank':
zpool scrub tank

# Check scrub status:
zpool status tank
```

TrueNAS schedules weekly scrubs by default. Run a scrub before and after major operations (adding disks, replacing a disk, large data migrations). A scrub after disk replacement confirms the resilver completed without silent errors.

## Copy-on-Write (CoW)

ZFS never modifies data in place. When a block is updated, ZFS:

1. Writes the new data to a free block.
2. Updates the pointer in the parent block to point to the new location.
3. The parent block's pointer update is itself a CoW write, propagating up the Merkle tree to the pool's uber-block (written last).

The old block is not freed until the transaction group commits. If power is lost mid-write, the old data is still intact — the partially-written new blocks are simply not pointed to by any committed state. ZFS requires no fsck after an unclean shutdown.

This also means ZFS has no "write hole" — the consistency problem in traditional RAID 5 where a partial stripe write leaves parity in an inconsistent state if power is lost. ZFS stripes are committed atomically at the transaction group boundary.

## Snapshots

A ZFS snapshot is a read-only point-in-time copy of a dataset. Because ZFS uses CoW, a snapshot at creation time is free — it contains no data, only references to the current block tree.

As the live dataset changes, CoW preserves the blocks that snapshots reference. Space consumption grows as the live dataset diverges from the snapshot. Deleting a snapshot frees its uniquely-referenced blocks.

### Snapshot Operations

```bash
# Create a snapshot:
zfs snapshot tank/data@2026-03-08

# List snapshots:
zfs list -t snapshot

# Roll back to a snapshot (destroys all changes since):
zfs rollback tank/data@2026-03-08

# Clone a snapshot into a writable dataset:
zfs clone tank/data@2026-03-08 tank/data-clone

# Destroy a snapshot:
zfs destroy tank/data@2026-03-08
```

TrueNAS manages snapshots via periodic snapshot tasks in the UI. A common policy: hourly snapshots for 24 hours, daily for 30 days, weekly for 52 weeks.

## Replication

ZFS send/receive streams snapshot data from one ZFS system to another. Only blocks that have changed since the last sent snapshot travel over the wire — incremental replication is highly bandwidth-efficient.

TrueNAS Replication Tasks automate this process over SSH with configurable schedules and retention policies on the remote side.

Replication is used for:
- **Offsite DR**: replicate to a second TrueNAS at a remote site.
- **Backup**: replicate to a TrueNAS with more storage for long-term retention.
- **Migration**: move datasets to new hardware by replicating then failing over.

## Compression

ZFS compression is applied transparently at the block level before writing to disk. Compression is dataset-level and takes effect on new writes — existing data is not retroactively compressed.

| Algorithm | Ratio | CPU Cost | Recommended For |
|-----------|-------|----------|-----------------|
| LZ4 | Moderate | Near-zero | Default — always enable |
| ZSTD | High | Low–Medium | General purpose, good balance |
| ZSTD-fast | Moderate | Very low | High-throughput workloads |
| GZIP | Very high | High | Cold archives, infrequently accessed data |
| off | None | None | Pre-compressed data (media files, already-compressed archives) |

**Always enable at minimum LZ4 compression.** For most workloads, LZ4 compression improves both storage efficiency and throughput — the CPU overhead is negligible, and less data written to disk means fewer I/O operations.

## ARC — Adaptive Replacement Cache

The ARC is ZFS's primary read cache, residing entirely in RAM. ZFS will claim as much free RAM as available for ARC up to configurable limits. The ARC uses a scan-resistant replacement algorithm that retains recently used and frequently used blocks.

### ARC Sizing Rules

| Pool Size | Minimum RAM | Recommended RAM |
|-----------|-------------|-----------------|
| < 10 TB | 8 GB | 16–32 GB |
| 10–50 TB | 16 GB | 32–64 GB |
| 50–100 TB | 32 GB | 64–128 GB |
| > 100 TB | 64 GB | 128 GB+ |

**ECC RAM is required for TrueNAS.** ZFS's CoW and checksumming guarantee data integrity end-to-end — but if RAM is silently corrupting data in the ARC before it reaches the disk, ZFS cannot detect it. A memory bit-flip in the ARC can write corrupt data with a valid checksum. ECC RAM detects and corrects single-bit errors, completing the integrity guarantee.

---

## L2ARC — Second-Level Read Cache

An L2ARC is an SSD (or set of SSDs) used as a secondary read cache beyond ARC. The ARC tracks which blocks were evicted from RAM; those blocks are candidates for L2ARC.

**L2ARC is frequently misunderstood.** It only helps when the hot working set exceeds available RAM. If ARC hit rates are already high, an L2ARC adds no value and slightly degrades performance (ARC must track L2ARC metadata). L2ARC data is lost on pool import (it is rebuilt gradually), so it does not substitute for RAM.

Use L2ARC when: you have identified that the working set exceeds ARC, you have budget for SSDs but cannot add RAM, and the workload is read-heavy.

## SLOG — Separate Log Device (ZIL)

The ZFS Intent Log (ZIL) records synchronous writes before acknowledging them to the client. By default, ZIL uses disk space from the pool itself. A separate log device (SLOG) places the ZIL on a dedicated SSD or NVMe drive.

A SLOG dramatically improves synchronous write performance — critical for:
- **NFS with sync writes** (VMware NFS datastores default to sync)
- **iSCSI** (databases typically request synchronous writes)
- **SMB with oplocks and sync semantics**

Without a SLOG, synchronous writes must wait for a full ZFS transaction group commit (~5 seconds by default). With a SLOG, the write is acknowledged as soon as it hits the fast NVMe SLOG, typically in microseconds.

**Mirror your SLOG.** A SLOG failure during a write means uncommitted synchronous writes are lost — equivalent to an unclean shutdown. Two SLOG devices in a mirror provide both performance and protection. Use enterprise-grade NVMe with power-loss protection (PLP) — consumer SSDs without PLP can lose buffered writes on power loss.

## Recommended Hardware Principles

| Component | Recommendation | Reason |
|-----------|---------------|--------|
| RAM | ECC, minimum 8 GB + 1 GB per TB | Data integrity; ARC performance |
| OS drive | Dedicated SSD or USB (mirrored) | Keep OS off data pool |
| Data drives | Enterprise SATA/SAS HDDs or SSDs | URE rates, vibration tolerance |
| NIC | Dedicated 10 GbE+ (separate from IPMI) | Storage traffic isolation |
| SLOG | Enterprise NVMe with PLP, mirrored | Sync write performance and safety |
| HBA | IT-mode HBA (LSI 9300 series) | Pass drives directly to ZFS |

**Use an HBA in IT mode (passthrough), not a RAID controller.** A RAID controller will attempt to manage drives itself. ZFS must own the drives directly to see SMART data, manage individual disk failures, and perform accurate checksumming. Flashing an LSI 9300/9400 series card to IT mode (non-RAID firmware) is the standard approach.
