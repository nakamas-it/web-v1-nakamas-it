## The Core Question: What Does the Client See?

Storage architecture decisions hinge on one question: what abstraction does the client receive?

- **A folder** on a network path → file storage (NAS)
- **A raw disk** that the client formats and owns → block storage (SAN)
- **A blob** identified by a key, accessed over HTTP → object storage

Everything else — hardware, protocols, vendors — flows from this distinction. The wrong choice for a workload does not just underperform; it can make certain operations impossible or impractical.

---

## File Storage — NAS

A Network Attached Storage device manages a full filesystem internally and exports it as a shared directory tree. The client mounts a network path. The filesystem lives on the server; the client is a consumer of it.

### Protocols

**SMB (Server Message Block)**: the native Windows file sharing protocol. Also called CIFS (Common Internet File System — an older dialect). Cross-platform support via Samba on Linux. Used for Windows file shares, home directories, DFS namespaces, printer sharing.

**NFS (Network File System)**: the UNIX/Linux standard. NFSv3 is stateless; NFSv4 adds stateful sessions, better security (Kerberos integration), and compound operations. Default for VMware VMFS-over-NFS datastores and Linux home directory sharing.

**AFP (Apple Filing Protocol)**: legacy Apple protocol. Deprecated in favour of SMB since macOS 10.9. TrueNAS still supports it for older Mac environments.

### Characteristics

- The filesystem (NTFS, ZFS, ext4, APFS) lives on the NAS, not on the client.
- Multiple clients can mount simultaneously with file-level locking (SMB opportunistic locks, NFS advisory locks).
- Easy to provision: create a share, set permissions, mount on clients.
- Performance scales with NAS CPU, RAM, network, and disk I/O. Latency is inherently higher than local storage due to network round-trips.
- Not suitable for workloads requiring low-latency, direct-disk access (SQL Server data files, Exchange databases, Hyper-V VHDs without specific configuration).

## Block Storage — SAN

A Storage Area Network presents raw block devices (Logical Unit Numbers — LUNs) to clients. The client's operating system receives what appears to be a local disk. The client formats it with whatever filesystem it chooses. The filesystem lives on the client, not on the storage array.

### Protocols

**iSCSI (Internet Small Computer Systems Interface)**: SCSI commands encapsulated in TCP/IP packets. Runs over standard Ethernet. No specialised hardware required (though a dedicated NIC and switch are strongly recommended). Most common for SMB/midmarket SAN deployments.

**Fibre Channel (FC)**: a dedicated storage networking fabric running at 8/16/32/64 Gbps. Not IP-based — a separate Layer 2 fabric (FC switches from Brocade/Broadcom or Cisco MDS). Requires Host Bus Adapters (HBAs) in servers instead of NICs. Low latency, high throughput, purpose-built for storage. Standard for enterprise environments with high I/O requirements.

**FCoE (Fibre Channel over Ethernet)**: FC frames encapsulated over 10/25 GbE. Converges storage and data networking on a single cable. Requires Data Center Bridging (DCB/CEE) on switches. Less common than pure iSCSI or pure FC but used in converged infrastructure.

**NVMe-oF (NVMe over Fabrics)**: NVMe commands over RDMA (RoCE or InfiniBand) or Fibre Channel. Eliminates the SCSI translation layer for all-flash arrays. Sub-100µs latency. The direction of high-performance block storage.

### Characteristics

- One client mounts the LUN read-write at a time (unless clustering software — MSCS, WSFC, Oracle RAC — manages shared-disk access).
- The SAN array is agnostic about what is stored — it sees only blocks.
- Higher performance than NAS for latency-sensitive workloads.
- More operationally complex: zoning, LUN masking, multipathing.
- Preferred for: SQL Server/Oracle databases, Exchange, Hyper-V VHDs, VMware VMFS datastores.

### FC Zoning

Fibre Channel fabrics use zoning to control which initiators (server HBAs) can see which targets (storage array ports). Analogous to VLANs in Ethernet networking.

- **Hard zoning** (World Wide Port Name — WWPN): enforced by the FC switch. Devices not in the same zone cannot communicate even if physically connected.
- **Soft zoning** (World Wide Node Name — WWNN): enforced in software. Less common, considered less secure.
- Best practice: single-initiator zones — one server HBA per zone, paired with the storage ports it needs. Never put two initiators in the same zone.

### LUN Masking

Even within a zone, LUN masking at the storage array level restricts which initiator can see which LUN. Provides a second layer of access control beyond zoning.

## Object Storage

Object storage is a flat namespace of blobs — binary objects of arbitrary size — each identified by a unique key and accompanied by extensible metadata. There is no directory hierarchy (though key prefixes can simulate one). Objects are accessed via REST APIs, not mounted filesystems.

### Characteristics

- **Immutable writes**: objects are written once and replaced atomically. No partial updates (with the exception of multipart uploads and range operations).
- **Metadata**: each object carries arbitrary key-value metadata. Enables rich cataloguing without a separate database.
- **Scale**: designed for horizontal scale-out to petabytes and billions of objects across commodity hardware.
- **No direct mounting**: you cannot mount an S3 bucket as a POSIX filesystem and expect POSIX semantics. FUSE-based mounts (s3fs, goofys) exist but are slow and do not support file locking.
- **Eventual consistency vs strong consistency**: S3 (since December 2020) provides strong read-after-write consistency for all operations. MinIO and Ceph also provide strong consistency.

### Protocols

- **S3 API (Amazon Simple Storage Service)**: the de facto standard. All major object stores implement S3 compatibility.
- **Swift (OpenStack Object Storage)**: used in OpenStack environments.
- **Azure Blob Storage**: Microsoft's object store, with its own API and S3 compatibility layer.

### Use Cases

Object storage is appropriate for: backups and archives, static web assets (images, video, JS bundles served via CDN), ML training datasets, log aggregation, compliance archives, any workload that writes once and reads many times, or writes at low frequency.

It is not appropriate for: databases, frequently updated files requiring locking, applications expecting POSIX filesystem semantics.

## Comparison Table

| Dimension | File (NAS) | Block (SAN) | Object |
|-----------|-----------|-------------|--------|
| Protocol | SMB, NFS | iSCSI, FC, NVMe-oF | S3, Swift |
| Filesystem location | On the NAS | On the client | No filesystem |
| Multi-client access | Yes (file locking) | No (without clustering) | Yes (API, no locking) |
| Latency | Medium | Low | High (API overhead) |
| Throughput | Medium–High | High | High (parallel GETs) |
| Ease of use | High | Medium | High (API), Low (POSIX) |
| Scalability | Vertical/limited scale-out | Vertical | Horizontal (petabyte scale) |
| Use cases | File shares, home dirs, VMware NFS | Databases, VMs, Exchange | Backups, media, archives, ML data |

## NAS + SAN Unified Arrays

Enterprise storage arrays (NetApp ONTAP, Dell PowerStore, Pure Storage FlashArray//C) present both NAS (NFS/SMB) and SAN (iSCSI/FC) protocols from the same hardware. The underlying storage pool is shared; different LUNs or volumes are exported via different protocols based on workload needs.

This "unified storage" model is common in midmarket and enterprise data centres — a single array serves databases via FC, VMware via NFS, and file shares via SMB.

## Hyperconverged Infrastructure (HCI)

HCI collapses storage, compute, and networking into a single commodity server layer. Local disks on each server are pooled into a distributed, software-defined storage fabric that presents block or file storage back to the same servers.

| Platform | Storage Software | Notes |
|----------|-----------------|-------|
| VMware vSAN | vSAN (VMkernel) | Tightly integrated with vSphere |
| Microsoft S2D | Storage Spaces Direct (Windows) | Windows Server 2016+ |
| Nutanix AOS | Nutanix Distributed Storage Fabric | Proprietary, hardware-agnostic |
| Ceph | Ceph RBD / CephFS / RGW | Open-source, all three storage types |

HCI eliminates dedicated storage hardware but couples storage scaling to compute scaling — you must add compute nodes to add storage capacity, even if you only need more storage.

## Choosing the Right Model

| Workload | Recommended Model | Reasoning |
|----------|------------------|-----------|
| SQL Server, Oracle | Block (SAN) | Requires low-latency random I/O, direct LUN access |
| File shares, home directories | File (NAS) | Multi-user access, directory hierarchy, easy management |
| VMware VMFS datastores | Block (iSCSI or FC) | VMFS requires block access |
| VMware NFS datastores | File (NFS) | Supported, simpler management |
| Hyper-V VHDs | Block or SMB 3.0 | SMB 3.0 supports VHD hosting with adequate performance |
| Backups | Object or File (NAS) | Large sequential writes, long retention |
| ML training data | Object | Billions of small files, parallel reads |
| Static web assets | Object (S3 + CDN) | Cost, scale, CDN integration |
| Compliance archive | Object | Immutability, retention locks, metadata |

The wrong choice is not always catastrophic, but it creates friction: a database on NAS latency degrades transaction throughput; an ML dataset in a filesystem creates inode exhaustion and metadata bottlenecks at scale; a shared filesystem used as object storage lacks the atomicity and horizontal scalability the workload demands.
